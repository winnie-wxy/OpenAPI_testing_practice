3. Authentication & Authorization

MethodHow It WorksSecurity LevelTesting Notes
Basic Auth Authorization: Basic base64(user:pass) Low Test with invalid/empty/missing credentials
API Key Key in header, query param, or cookie Medium Test key rotation, revoked keys, rate limits per key
Bearer Token (JWT) Authorization: Bearer <token> High Test expired tokens, tampered payload, missing claims

CoverG Auth Pattern

# Session-scoped: one auth call per test run
@pytest.fixture(scope="session")
def auth_token(base_url, auth_credentials):
    username, password = auth_credentials
    client = BookingClient(base_url)
    response = client.create_token(username, password)
    return response.json()["token"]

# Client with self-healing token refresh
@pytest.fixture(scope="session")
def client(base_url, auth_token, auth_credentials):
    username, password = auth_credentials
    return BookingClient(
        base_url, token=auth_token,
        username=username, password=password,  # enables auto-refresh on 403
    )

4. API Communication Patterns

Polling

Client repeatedly checks for updates at intervals.

  • Simple to implement
  • Wastes bandwidth
  • Latency = poll interval

Use case: CI status checks

Webhooks

Server pushes updates to a registered callback URL.

  • Real-time, efficient
  • Harder to test (need listener)
  • Delivery guarantees matter

Use case: Payment notifications

Event-Driven

Message broker (Kafka, RabbitMQ) decouples producers and consumers.

  • Async, scalable
  • Test ordering, replay, dead-letter
  • Schema registry for contracts

Use case: Claims pipeline

Testing Each Pattern

PatternWhat to TestKey Challenges
PollingTimeout, frequency, data freshnessEventual consistency, race conditions
WebhooksDelivery, retries, idempotency, signature validationNeed mock receiver, ordering
EventsMessage schema, ordering, dead-letter, replayAsync timing, consumer lag
CoverG: Polling pattern used in flaky test fix — test_create_and_verify_with_polling retries GET with backoff instead of assuming immediate consistency. Webhook delivery is a documented P1 gap (API doesn't support it).

5. Test Architecture & Strategy

Four-Layer API Test Architecture

LayerPurposeCoverG CountCI Stage
Smoke Is the API alive? Auth works? Basic CRUD? 6 tests Gate (blocks all)
Regression Full lifecycle, negative cases, data variations, filtering 15 tests Gate (blocks NFR)
Contract Pydantic schema validation — catches field drift 3 tests With regression
Non-Functional Performance P95, concurrency, security, resilience, self-healing 36 tests Final stage

Test Design Strategy

What to Test

  • Happy path — standard CRUD success
  • Negative cases — invalid input, missing auth, not found
  • Boundary values — empty strings, max lengths, zero
  • Data variations — unicode, special chars, apostrophes
  • Concurrency — parallel writes, read-under-write
  • Security — injection, auth bypass, oversized payloads

What NOT to Test (in API tests)

  • UI rendering or browser behaviour
  • Database internals (test via API contracts)
  • Third-party API internals (mock at boundary)
  • Implementation details (test behaviour, not code)
  • Exhaustive combinations (use risk-based priority)

Yield Fixtures — Guaranteed Cleanup

@pytest.fixture
def created_booking(client):
    payload = build_booking_payload()
    response = client.create_booking(payload)
    booking_id = response.json()["bookingid"]
    yield booking_id, payload          # test runs here
    client.delete_booking(booking_id)  # cleanup runs even on failure

Factory Fixtures — Flexible Setup

@pytest.fixture
def make_booking(client):
    created_ids = []
    def _make(**overrides):
        payload = build_booking_payload(**overrides)
        booking_id = client.create_booking(payload).json()["bookingid"]
        created_ids.append(booking_id)
        return booking_id, payload
    yield _make
    for bid in created_ids:
        client.delete_booking(bid)

Markers & Selective Execution

7. Self-Healing Test Patterns NEW

Self-healing patterns make tests resilient to transient failures without hiding real bugs.

Auto-Retry with Backoff

Client retries on 5xx/429/ConnectionError with exponential backoff. Does NOT retry 4xx (real failures).

# Retryable: 429, 500, 502, 503, 504
# Backoff: 0.5s, 1s, 2s (configurable)
# Max retries: 3 (configurable)
client = BookingClient(
    base_url,
    max_retries=3,
    backoff_base=0.5,
)

8 tests verify retry, exhaustion, backoff timing, non-retry on 4xx.

Token Refresh on 403

When a PUT/PATCH/DELETE gets 403, client re-authenticates and retries once. Does NOT refresh on GET (403 on GET = real auth issue).

# Token refresh is "free" —
# doesn't consume retry budget
client = BookingClient(
    base_url, token=auth_token,
    username="admin",
    password="password123",
)

5 tests verify refresh, skip on GET, no-creds guard, single-attempt limit.

Schema Tolerance

Pydantic models use extra="allow" — tolerates new fields from API changes without breaking existing assertions.

class Booking(BaseModel):
    model_config = ConfigDict(extra="allow")
    firstname: str  # required fields
    # ... new fields accepted silently

4 tests verify extra fields accepted, required fields still enforced.

Stale Data Detection

Precondition checks verify data exists before asserting on it. Catches phantom passes from eventual consistency.

# Verify booking exists BEFORE update
get_resp = client.get_booking(booking_id)
assert get_resp.status_code == 200, \
    f"Precondition: booking {booking_id} missing"
# Now safe to update
update_resp = client.update_booking(...)

2 tests verify preconditions and post-deletion staleness.

8. Flaky Test Debugging

Git history tells the story: introduce anti-patterns → diagnose → fix.

Anti-PatternWhy It FlakesFix
Race Condition
Create → immediate GET		 Eventual consistency — data not queryable yet Retry/poll with backoff
Shared State
Module-level global variable		 Breaks with parallel execution or random order Per-test fixture isolation
Timing Assertion
assert elapsed < 500ms		 Network jitter, CI runner contention Multi-sample P95 statistical approach
"Reruns are a safety net, not a solution. If a test reruns in >20% of builds, it needs fixing — not more reruns. I track rerun rate as a quality signal."

Debugging Workflow

  1. Reproduce: pytest --count=10 -x tests/path/to_test.py (run 10x, stop on first fail)
  2. Isolate: pytest -k "test_name" --no-header -rN (single test, minimal output)
  3. Check parallelism: pytest -n 4 --dist=loadfile (does it only fail in parallel?)
  4. Check ordering: pytest -p randomly (does it only fail in certain order?)
  5. Root cause: Usually shared state, timing, or external dependency
  6. Fix + verify: Run 20x in CI to confirm stability

10. Key API Testing Concepts

ConceptWhat It IsWhat to Test
Idempotency Same request N times = same result (GET, PUT, DELETE are idempotent; POST is not) Send duplicate POST — does it create duplicates? Does the API support idempotency keys?
Rate Limiting Server limits requests per time window (429 response) Exceed limit, verify 429, check Retry-After header, verify recovery after window
Pagination Large result sets returned in pages (offset/limit, cursor, keyset) First/last page, empty page, page size boundary, consistency during writes
Versioning URL path (/v2/) or header (Accept: application/vnd.api+json;version=2) Old version still works, new version has new fields, invalid version returns error
CORS Browser security: server controls which origins can call the API Preflight OPTIONS, Access-Control-Allow-Origin header, credentials mode
Contract Testing API response matches agreed schema (Pydantic, JSON Schema, Pact) Required fields present, types correct, unknown fields handled, breaking changes detected
Retry & Backoff Client retries transient failures with increasing delays Retries on 5xx, exponential backoff timing, max retry limit, no retry on 4xx
CoverG known gaps: Idempotency (GAP-002), Rate Limiting (GAP-003), Pagination (GAP-005) — documented with rationale in the coverage matrix rather than pretending they don't exist.

11. Coherency Check System

The most dangerous test suite passes 100% and gives you false confidence. Coherency checks tell you what you're NOT testing.

Layer 1: Structural (Automated)

Python script parses coverage/matrix.yaml and validates against filesystem.

Violations: 0 errors, 3 warnings

Checks: covered_without_tests, missing_test_file, missing_test_function, orphaned_test, marker_mismatch, p1_gap

Layer 2: Semantic (AI-Assisted)

Claude Code skill analyses actual test code quality:

  • Assertion quality — does the test actually validate intent?
  • Negative case coverage — P1 reqs have both happy + failure paths?
  • Contract drift — Pydantic model vs live API shape
  • Data isolation — no shared state, no order dependency

Invoked via /test-coherency skill.

13. Interview Q&A

Q: How do you decide what to automate vs test manually?

Automate the regression safety net — things that must pass every release. Manual test for exploratory, new feature validation, and anything involving subjective judgment. The split shifts over time — what starts manual gets automated once the feature stabilises.

Q: How do you handle test data in a multi-tenant system?

Factory pattern with randomised data — each test creates its own tenant, policies, claims. Yield fixtures handle cleanup. No shared state between tests, no ordering dependencies. Essential for parallel execution with pytest-xdist.

Q: How do you prioritise what to test with limited resources?

Risk-based: impact × likelihood. P1 is anything that blocks revenue or violates compliance. P2 is customer-facing degradation. P3 is internal tooling. I document gaps explicitly — known untested areas with rationale — rather than pretending everything is covered.

Q: Tell me about a time you dealt with flaky tests in CI.

I have a concrete example in this project. Three anti-patterns: race condition (create → immediate GET), shared state (module-level variable), single-sample timing assertion. Each has a specific fix: polling with backoff, per-test fixture isolation, multi-sample P95 statistical approach. Reruns are a safety net, not a solution — I track rerun rate as a quality signal.

Q: How do you use AI in your QA work?

Three levels: (1) Code generation — AI writes the boilerplate, I review. (2) Workflow automation — custom skills for test plans, Jira cards, coverage analysis. (3) Strategic analysis — AI analyses requirements and identifies test gaps. The key is human-in-the-loop — AI accelerates, humans validate.

Q: How would you onboard a junior test engineer?

Start with the patterns — show them the project structure, explain why each layer exists. Pair on the first few tests. The flaky test commit history is onboarding material — intentional anti-patterns, then fixes. Graduate to writing tests independently with PR review. CLAUDE.md and skills give guardrails so they follow team patterns.

Q: What's your experience with Postman?

I use Postman for API exploration and debugging — faster than writing a test when investigating behaviour. This project includes a Postman collection mirroring the pytest suite. In a team setting, the collection is the shared reference — anyone can import it and explore the API without the test framework.

Q: How do you measure test quality?

Four signals: (1) Defect escape rate — bugs reaching production despite passing tests. (2) Flake rate — tests needing reruns. (3) Coverage coherency — requirements with no tests (my coherency checker). (4) Time to feedback — how fast CI gives results. Passing tests is table stakes — these metrics tell you if the tests are doing their job.

Q: What makes a good API test vs a bad one?

Good: tests behaviour not implementation, isolated data, meaningful assertions (not just status code), documents what it validates. Bad: depends on execution order, shares state, asserts only response.status_code == 200, no cleanup, hardcoded IDs.

Q: How do you handle API versioning in tests?

Base URL includes version (/v1/, /v2/), configured via environment. Contract tests validate both versions during migration. Schema tolerance (extra="allow" in Pydantic) prevents new fields from breaking existing tests. The coherency checker catches stale tests after API changes.

Q: What's your approach to performance testing APIs?

Multi-sample statistical approach, not single assertions. Take 5-10 samples per endpoint, assert on P95 (not mean/max). Use ThreadPoolExecutor for concurrent load tests — verify no 5xx under 10 concurrent users. Performance baselines per environment, tracked over time. For serious load testing, Locust with proper connection pooling.

Q: How do you test microservice dependencies?

Contract testing at service boundaries — each service validates its schema independently. Mock external dependencies at the API client layer. Integration tests verify actual inter-service communication in staging. Use circuit breaker patterns — test what happens when a dependency is down (resilience tests).
